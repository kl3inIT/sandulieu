<threat_model>

## Trust Boundaries

| Boundary                                                 | Description                                                                 |
| -------------------------------------------------------- | --------------------------------------------------------------------------- |
| admin route → organization feature hooks                 | Untrusted route params and user form input cross into data operations here  |
| organization API mock layer → department dependency data | Delete decisions depend on child-record reads from another mock data source |
| mutation result → query cache                            | Incorrect invalidation can surface stale or deleted records in admin UI     |

## STRIDE Threat Register

| Threat ID | Category | Component                                                   | Disposition | Mitigation Plan                                                                                                                                   |
| --------- | -------- | ----------------------------------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-02-01   | T        | `shared/api/organization.api.ts` create/update/delete paths | mitigate    | Validate input through the shared Zod-backed form contract and re-check id/code constraints in API functions before mutating mock data            |
| T-02-02   | I        | organization detail contract                                | mitigate    | Return only organization summary and metadata fields needed for admin UI; do not fabricate unrelated sensitive fields                             |
| T-02-03   | R        | delete guard and soft-delete behavior                       | mitigate    | Return structured guard results with explicit reason keys and dependency counts so list/detail surfaces render auditable, deterministic messaging |
| T-02-04   | D        | query invalidation after mutation                           | mitigate    | Invalidate targeted list/detail keys after create/update/delete and avoid broad stale cache reuse                                                 |
| T-02-05   | E        | route identity vs editable business fields                  | mitigate    | Keep `organizationId` route identity bound to stable `id`; never use mutable `name` or `code` as detail cache identity                            |

</threat_model>
