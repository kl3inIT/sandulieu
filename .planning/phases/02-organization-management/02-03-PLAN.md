<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| form input → organization mutations | User-controlled values become create/update payloads |
| route param → edit query | Untrusted `organizationId` selects which record is editable |
| mutation completion → redirect | Redirect target depends on stable returned record identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | T | create/edit form submission | mitigate | Reuse Zod-backed form validation from the feature layer and surface normalized mutation errors back into the route UI |
| T-02-11 | E | edit route identity | mitigate | Load and update records by stable `organizationId` only; do not derive editable target from mutable name/code |
| T-02-12 | R | create/update outcomes | mitigate | Keep success flow deterministic by redirecting to the returned detail route and invalidating detail/list queries from the mutation layer |
| T-02-13 | D | edit route loading path | mitigate | Use detail query hydration instead of ad hoc duplicate fetches and render loading/error states explicitly |
</threat_model>
