<threat_model>

## Trust Boundaries

| Boundary                                           | Description                                                                                     |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| admin route → department feature hooks             | Untrusted route params and user form input cross into organization-scoped department operations |
| department API mock layer → member dependency data | Delete decisions and member summary depend on child-record reads from another mock data source  |
| mutation result → query cache                      | Incorrect invalidation can surface stale, cross-organization, or deleted records in admin UI    |

## STRIDE Threat Register

| Threat ID | Category | Component                                                 | Disposition | Mitigation Plan                                                                                                                                     |
| --------- | -------- | --------------------------------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-03-01   | T        | `shared/api/department.api.ts` create/update/delete paths | mitigate    | Validate input through the shared Zod-backed form contract and re-check id/code constraints within the organization scope before mutating mock data |
| T-03-02   | I        | department detail contract                                | mitigate    | Return only department summary, parent context, and member-summary fields needed for admin UI; do not fabricate unrelated detail-only fields        |
| T-03-03   | R        | delete guard and soft-delete behavior                     | mitigate    | Return structured guard results with explicit reason keys and dependency counts so list/detail surfaces render deterministic messaging              |
| T-03-04   | D        | query invalidation after mutation                         | mitigate    | Invalidate targeted organization-qualified list/detail keys after create/update/delete and avoid flat cache reuse                                   |
| T-03-05   | E        | nested route identity vs mutable business fields          | mitigate    | Keep nested route identity bound to stable `organizationId` + `departmentId`; never use mutable `name` or `code` as detail cache identity           |

</threat_model>
