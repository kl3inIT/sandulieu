<threat_model>

## Trust Boundaries

| Boundary                                 | Description                                                                   |
| ---------------------------------------- | ----------------------------------------------------------------------------- |
| nested form input → department mutations | User-controlled values become create/update payloads                          |
| route params → edit query                | Untrusted `organizationId` and `departmentId` select which record is editable |
| mutation completion → redirect           | Redirect target depends on stable returned nested identity                    |

## STRIDE Threat Register

| Threat ID | Category | Component                   | Disposition | Mitigation Plan                                                                                                                                |
| --------- | -------- | --------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| T-03-10   | T        | create/edit form submission | mitigate    | Reuse Zod-backed form validation from the feature layer and surface normalized mutation errors back into the route UI                          |
| T-03-11   | E        | nested edit route identity  | mitigate    | Load and update records by stable `organizationId` + `departmentId` only; do not derive editable target from mutable name/code                 |
| T-03-12   | R        | create/update outcomes      | mitigate    | Keep success flow deterministic by redirecting to the returned nested detail route and invalidating organization-qualified detail/list queries |
| T-03-13   | D        | nested edit loading path    | mitigate    | Use detail query hydration instead of ad hoc duplicate fetches and render loading/error states explicitly                                      |

</threat_model>
